# Таблица товаров для «Заполнителя» (Альта-Софт)

Веб-приложение для специалиста по таможенному оформлению. Состоит из двух разделов:

1. **Таблица товаров** — из загруженных коммерческих документов (инвойс, упаковочный лист, спецификация) собирает единую таблицу товаров для последующего импорта в программу «Заполнитель» разработчика Альта-Софт.
2. **Проверка ограничений** — сверяет коды ТН ВЭД с перечнями постановлений Правительства РФ 311, 312 и 313.

Первый раздел работает целиком в браузере: документы **не отправляются ни на какой сервер**. Второй раздел загружает перечни кодов с сайта alta.ru через прокси — сами товарные файлы при этом никуда не уходят, обработка остаётся локальной.

## Правила обработки

- **Поиск данных** ведётся по ключевым словам в заголовках колонок и тексте документов: *наименование, название, модель, артикул, изделие, код изделия, вес нетто, вес брутто, цена, стоимость, количество, ед. изм., грузовое место* и их англоязычные аналоги (*description, article, model, net weight, gross weight, price, amount, qty, place, box* и т. п.).
- **Коды единиц измерения** проставляются по классификатору ОКЕИ (шт → 796 / ШТ, кг → 166 / КГ, компл → 839 / КОМПЛ, пар → 715 / ПАР и др.). Если единица не распознана, соответствующие графы остаются пустыми.
- **Брутто грузового места** распределяется между товарами этого места **пропорционально их весу нетто**, с округлением до трёх знаков после запятой (остаток округления относится на последний товар места, поэтому сумма всегда сходится с брутто места).
- **Повторяющиеся товары** (одинаковое наименование и артикул) объединяются в одну строку; количество, стоимость и веса суммируются. Если товар лежит в нескольких местах, номера мест перечисляются через запятую.
- **Режим «построчно»** — каждая строка упаковочного листа становится отдельной строкой таблицы, цена подтягивается из инвойса по артикулу.
- **Расхождения между документами** по количеству, весу или стоимости выделяются **жёлтой заливкой**; математические расхождения (расчёт не сходится со значением в файле) — **розовой**.
- **Недостающие данные** не выдумываются: заполняются только найденные графы, остальные остаются пустыми.
- Строки «Итого / Всего / Total» из документов в таблицу не попадают.

## Поддерживаемые форматы

| Формат | Качество распознавания |
|--------|------------------------|
| XLSX, XLS, CSV | Высокое — таблицы читаются напрямую |
| DOCX, TXT | Хорошее, если данные оформлены таблицей или строками с разделителями |
| PDF | Текстовые PDF читаются построчно; **сканы без текстового слоя не распознаются** (нужен предварительный OCR) |

## Проверка работы

В папке `examples/` лежат тестовые файлы — `invoice_primer.xlsx` и `packing_list_primer.xlsx`. Загрузите оба: приложение объединит дубликаты «Фильтр масляный», распределит брутто места № 1 (150 кг) пропорционально нетто насоса и прокладки (139,535 + 10,465) и выделит жёлтым строку «Прокладка резиновая» — по ней количество в инвойсе (100) расходится с упаковочным листом (90).

# Раздел 2. Проверка ограничений по постановлениям 311, 312, 313

Сверяет коды ТН ВЭД из таблицы товаров с перечнями постановлений Правительства РФ
[311](https://www.alta.ru/tamdoc/22ps0311/),
[312](https://www.alta.ru/tamdoc/22ps0312/) и
[313](https://www.alta.ru/tamdoc/22ps0313/)
и возвращает исходную таблицу с добавленными графами.

**Это инструмент предварительной проверки, а не замена работе с первоисточником.** Перечни меняются часто: только за последний год правки вносили постановления 1374, 479, 599, 1516, 257 и 404. Перед подачей ДТ сверяйте позиции с действующей редакцией.

## Что на входе

Таблица товаров в XLSX, XLS или CSV. Обязательна графа с кодом ТН ВЭД.

Подойдёт и таблица, собранная в первом разделе: графа «Код ТН ВЭД ЕАЭС» распознаётся автоматически.

### Как определяются границы таблицы

В реальных спецификациях таблица редко начинается с первой строки: сверху бывает название документа, номер, дата, подпись «Список товаров», а снизу — «Итого», сводные веса и количество мест.

Приложение ищет **строку заголовков** — первую строку, где встречается название графы с кодом ТН ВЭД («Код ТН ВЭД», «ТН ВЭД», «HS code», «Commodity code»). Всё, что выше, отбрасывается.

Товарами считаются строки, где в этой графе стоит код из шести и более цифр. Строки без кода — «Итого», разделители, сводные суммы — в результат не попадают. После обработки приложение сообщает, в какой строке нашло шапку, из какой графы взяло код и сколько строк отбросило: границы разбора всегда видны, доверять им вслепую не нужно.

### Если заголовка нет

Тогда графа определяется по содержимому. Это ненадёжно, и приложение ведёт себя осторожно.

Рядом с кодом ТН ВЭД почти всегда стоит артикул, который тоже выглядит как длинное число. Перепутать их — значит молча проверить не то и получить сплошные «нет». Поэтому:

- графы с заголовками «Артикул», «Item code», «Код изделия», «Part number», «SKU» исключаются из поиска;
- значения с буквами (`300513672E`) не считаются кодами;
- полные десятизначные коды дают преимущество;
- если несколько граф похожи одинаково, приложение **отказывается выбирать** и просит назвать графу явно.

Надёжнее всего назвать графу «Код ТН ВЭД» или «HS code» — тогда она определится точно.

## Что на выходе

Исходная таблица без изменений плюс семь граф:

| Графа | Значения |
| --- | --- |
| ПП 311 | да / нет / проверить |
| Искл. из 311 | да / нет / проверить |
| ПП 312 | да / нет / проверить |
| Искл. из 312 | да / нет / проверить |
| ПП 313 | да / нет / проверить |
| Искл. из 313 | да / нет / проверить |
| Итог проверки | словесный вывод |

Результат скачивается в XLSX. Ячейки с «да» подсвечены розовым, «проверить» — жёлтым. В примечании к ячейке постановления указано, какое приложение и какая позиция сработали. Второй лист книги содержит источник данных, дату проверки и памятку по чтению результата.

## Как читать результат

Пара граф работает так:

- **ПП = да, Искл. = нет** — товар подпадает под ограничение;
- **ПП = да, Искл. = да** — позиция выведена из-под ограничения, товар **не** подпадает;
- **ПП = нет** — совпадений в перечне не найдено.

Два «да» подряд читаются не как «двойной запрет», а ровно наоборот. Чтобы не путаться, ориентируйтесь на графу **«Итог проверки»** — она сводит все три постановления в одну фразу.

### Когда появляется «проверить»

Автоматика не угадывает там, где решение за декларантом. «Проверить» возникает в двух случаях:

1. **Код товара короче кода перечня.** В файле указано `8501`, а в перечне `8501 53 810 0`. Товарная группа шире позиции перечня — нужен полный код.
2. **Текстовое исключение.** В перечнях встречаются исключения без кодов, например «за исключением товаров, указанных в абзаце двадцать седьмом пункта 2». Такое исключение проверяется только вручную.

## Логика сопоставления

Коды нормализуются — пробелы убираются, остаются цифры: `8501 53 810 0` превращается в `8501538100`.

Совпадение определяется по префиксу: код товара должен начинаться с кода позиции перечня. Затем так же проверяются исключения этой позиции.

Пример. Товар `8501538100`, позиция перечня `8501 (за исключением 8501 20 000 9, 8501 52 200 1)`:

- `8501538100` начинается с `8501` → **ПП 311 = да**;
- ни с `8501200009`, ни с `8501522001` не начинается → **Искл. = нет**;
- итог: подпадает под 311.

### Приложения

У постановления 311 три приложения, у 312 — шесть, у 313 — несколько. Совпадение по **любому** приложению даёт «да» в общей графе. Какое именно приложение сработало, видно в примечании к ячейке в выгруженном XLSX и во всплывающей подсказке на экране.

Учтите юридическую разницу: приложение № 1 к постановлению 311 — это запрет на вывоз, а приложения № 2 и № 3 — разрешительный порядок. Общая графа их не различает, поэтому при «да» смотрите примечание.

## Откуда берутся перечни

Источники пробуются по очереди:

1. **alta.ru через публичный CORS-прокси.** Напрямую нельзя: у alta.ru нет заголовков CORS. Пробуется несколько прокси подряд.
2. **Локальный снимок `spisok.json`** из репозитория, если сайт не ответил.
3. **Таблица ограничений, загруженная вручную**, если нет ни сайта, ни снимка.

Приложение сообщает, какой источник сработал. Число найденных позиций выводится в блоке «Перечни кодов» — если оно резко упало, значит разметка сайта изменилась и парсер требует правки.

### Загрузка таблицы ограничений вручную

Если перечни недоступны, приложение просит загрузить актуальную таблицу и блокирует шаг с товарами до её получения. Кнопка «Загрузить таблицу ограничений» доступна и в обычном режиме — на случай, если хочется работать по своему файлу.

Формат файла: **XLSX с отдельным листом на каждое постановление**. Номер постановления берётся из названия листа, поэтому в нём должно встречаться число 311, 312 или 313 — например, «Постановление 311».

Структура каждого листа:

| Графа | Содержимое |
| --- | --- |
| A | Код ТН ВЭД |
| B | Наименование товара (необязательно) |
| C | — |
| D | Исключения — общий список кодов постановления |

Первая строка считается заголовком и пропускается. Коды могут храниться как числа. Если в одной ячейке слиплись два кода через точку или запятую, они разделяются.

Листы, которых нет в файле, помечаются как непроверенные: в соответствующей графе появится «нет данных», а в итоге — «нет перечня по 313». Это честнее, чем показать «нет ограничений» там, где проверка не проводилась.

### Про сводные таблицы из интернета

Готовые таблицы ограничений, которые ходят по сети, иногда сводят несколько перечней в один лист. Тогда товар получает «да» сразу по двум постановлениям, хотя в первоисточнике позиция есть только в одном.

Приложение проверяет пересечение листов и предупреждает, если оно превышает половину позиций. Такое предупреждение — повод сверить спорные позиции с alta.ru вручную.

Парсер отбрасывает:

- блоки после маркера «См. пред. ред.» — это старые редакции;
- позиции с текстом «Позиция исключена»;
- годы и номера документов, попадающие в текст исключений (`от 9 марта 2022 г. N 312` — здесь `2022` не код товара).

### Обновление снимка

Файл `spisok.json` в репозитории — **неполная заготовка**. Перед первым серьёзным использованием соберите полный снимок:

```bash
pip install requests beautifulsoup4
python3 sobrat_spisok.py
```

Скрипт создаёт `spisok.json` и печатает статистику по каждому приложению. Выборочно сверьте несколько позиций с сайтом перед коммитом.

Либо нажмите в приложении «Обновить перечни с alta.ru» — тогда данные возьмутся с сайта напрямую, минуя снимок.

## Ограничения раздела 2

- **Проверяется только код ТН ВЭД.** Постановления содержат исключения по происхождению товара, направлению вывоза, таможенной процедуре, назначению груза и статусу отправителя — ничего этого приложение не знает. «Нет» в таблице не означает, что ограничений нет вообще.
- Публичные прокси нестабильны. Если проверка нужна в конкретный день, обновите снимок заранее.
- Постановления действуют до 31 декабря 2027 года и не подлежат применению с 1 января 2028 года.

## Проверка работы раздела 2

Файл `examples/tovary_primer.xlsx` содержит четырнадцать строк, задевающих все ветки логики: обычное совпадение, оба кода из списка исключений 8501, товар без ограничений, слишком короткий код, пустой код и позицию с текстовым исключением (8905).

# Установка на GitHub Pages

1. Загрузите содержимое этой папки в репозиторий.
2. Откройте **Settings → Pages**.
3. В разделе **Build and deployment** выберите: Source — *Deploy from a branch*, Branch — `main`, папка — `/ (root)`. Нажмите **Save**.
4. Через одну–две минуты приложение будет доступно по адресу `https://<ваш-логин>.github.io/<имя-репозитория>/`.

Либо через командную строку:

```bash
git add .
git commit -m "Добавлен раздел проверки ограничений 311, 312, 313"
git push
```

Локально приложение можно открыть двойным щелчком по `index.html` — сервер не нужен (требуется доступ в интернет для загрузки библиотек с CDN).

# Общие ограничения

- Приложение — помощник декларанта, а не замена проверки: перед подачей ДТ сверяйте данные с первоисточниками.
- Справочник ОКЕИ включает наиболее употребимые единицы; при необходимости список легко расширить в начале файла `app.js` (массив `OKEI`).
- PDF со сложной вёрсткой таблиц могут распознаваться частично — в этом случае приложение сообщит, что данные в файле не найдены.
